构建 Agent 框架最难的部分之一，就是设计它的行动空间（action space）。

Claude 通过工具调用（Tool Calling）来行动，但在 Claude API 中有多种方式可以构建工具，比如使用 bash、技能（skills），以及最近新增的代码执行功能（想了解更多关于 Claude API 中程序化工具调用的内容，可以阅读 @RLanceMartin 的新文章）。

面对这么多选项，你该如何设计 Agent 的工具？是只需要一个工具（比如代码执行或 bash）就够了吗？还是你需要 50 个工具，每个对应 Agent 可能遇到的特定场景？

为了站在模型的角度思考，我喜欢想象自己被给了一道难题。为了解这道题，你需要什么工具？这取决于你自己的技能！

纸笔是最低限度，但你可能会被手工计算限制。计算器会更好，但你需要知道如何操作那些高级功能。最快最强大的选择是计算机，但你得懂得用它来编写和执行代码。

这是设计 Agent 的一个有用框架。你想给它的工具应该与其自身能力相匹配。但你怎么知道它的能力是什么呢？你需要留心观察、阅读它的输出、不断实验。你要学会像 Agent 一样思考。

以下是我们构建 Claude Code 过程中，通过观察 Claude 学到的一些教训。

## 提升引导能力（Elicitation）与 AskUserQuestion 工具

![](https://pbs.twimg.com/media/HCLxrfXbEAUXwRV.jpg)

构建 AskUserQuestion 工具时，我们的目标是提升 Claude 的提问能力（通常称为 elicitation）。

虽然 Claude 可以用纯文本提问，但我们发现回答这些问题让人感觉耗时过长。如何降低这种摩擦，提升用户与 Claude 之间的沟通带宽？

### 尝试 #1 - 修改 ExitPlanTool

我们首先尝试的是给 ExitPlanTool 添加一个参数，让计划旁边附带一组问题。这是最容易实现的方式，但它让 Claude 感到困惑，因为我们同时要它给出计划和一组关于计划的问题。如果用户的回答与计划内容冲突怎么办？Claude 需要调用两次 ExitPlanTool 吗？我们需要另一种方法。

（想了解更多关于我们为什么创建 ExitPlanTool，可以阅读我们关于提示缓存的文章）

### 尝试 #2 - 改变输出格式

接下来我们尝试修改 Claude 的输出指令，让它使用一种稍加修改的 markdown 格式来提问。比如，我们可以让它输出带括号选项的项目符号问题列表。然后我们可以解析这些问题并格式化为用户界面。

虽然这是最通用的修改方式，Claude 似乎也能输出这种格式，但结果并不稳定。Claude 会追加额外句子、省略选项，或者使用完全不同的格式。

### 尝试 #3 - AskUserQuestion 工具

![](https://pbs.twimg.com/media/HCLxg2JbsAA3Ag_.jpg)

最后，我们决定创建一个 Claude 可以随时调用的工具，特别是在规划模式（plan mode）时提示它调用。当工具触发时，我们会显示一个弹窗来展示问题，并阻塞 Agent 的循环直到用户回答。

这个工具让我们能够提示 Claude 输出结构化内容，并确保它给用户提供多个选项。它还让用户可以组合使用这个功能，比如在 Agent SDK 中调用，或在技能中使用。

最重要的是，Claude 似乎很喜欢调用这个工具，我们发现它的输出效果很好。即使再好的工具，如果 Claude 不懂如何调用，也是白搭。

这是 Claude Code 中 elicitation 的最终形态吗？我们不确定。正如下一个例子所示，对某个模型有效的东西未必对另一个模型最好。

## 随能力进化而更新 - Tasks 与 Todos

![](https://pbs.twimg.com/media/HCL0gcObkAA4tKt.jpg)

我们刚推出 Claude Code 时，意识到模型需要一个待办清单（Todo list）来保持专注。可以在开始工作时就写好待办，然后在模型完成工作时逐项勾选。为此我们给了 Claude TodoWrite 工具，用来编写或更新待办并显示给用户。

但即便如此，我们还是经常看到 Claude 忘记自己要做什么。为了适应，我们在每 5 轮对话后插入系统提醒，让 Claude 记住它的目标。

但随着模型的改进，它们不仅不再需要待办提醒，反而可能觉得待办清单是一种限制。被反复提醒待办清单会让 Claude 觉得必须坚持清单而不能修改它。我们还看到 Opus 4.5 在子 Agent 使用上有了很大提升，但子 Agent 如何在一个共享待办清单上协调呢？

观察到这点后，我们用 Task 工具替代了 TodoWrite 工具（更多关于 Tasks 的内容可以阅读这里）。如果说待办是为了让模型保持专注，那么任务更多是帮助 Agent 之间互相通信。任务可以包含依赖关系、在子 Agent 之间共享更新，模型也可以修改和删除它们。

随着模型能力的提升，你的模型曾经需要的工具现在可能会成为它们的束缚。不断重新审视之前关于需要哪些工具的假设是很重要的。这也是为什么专注于一小部分模型集合是有用的，这些模型的能力特征相对接近。

## 设计搜索界面

对 Claude 来说，特别重要的一组工具是可以用来构建自身上下文的搜索工具。

Claude Code 刚推出时，我们使用 RAG 向量数据库来为 Claude 找到相关上下文。虽然 RAG 强大且快速，但它需要索引和设置，并且在不同环境中可能很脆弱。更重要的是，上下文是给 Claude 的，而不是让它自己找到的。

但如果 Claude 能在网上搜索，为什么不能搜索你的代码库？通过给 Claude 一个 Grep 工具，我们可以让它自己搜索文件并构建上下文。

随着 Claude 变得更聪明，我们观察到一个模式：如果给它对路的工具，它会越来越擅长构建自己的上下文。

当我们引入 Agent Skills 时，正式提出了渐进式披露（progressive disclosure）的概念，允许 Agent 通过探索逐步发现相关上下文。

Claude 可以阅读技能文件，这些文件可以引用其他文件，模型可以递归阅读。事实上，技能的常见用途之一就是给 Claude 增加更多搜索能力，比如如何调用 API 或查询数据库的指令。

在一年时间里，Claude 从几乎无法构建自己的上下文，到能够跨多层文件进行嵌套搜索，找到它需要的精确上下文。

渐进式披露现在是我们不增加工具就能添加新功能的常用技术。

## 渐进式披露 - Claude Code Guide Agent

Claude Code 目前有约 20 个工具，我们不断问自己是否真的需要所有这些。添加新工具的门槛很高，因为这会让模型多一个需要考虑的选择。

例如，我们注意到 Claude 对如何使用 Claude Code 本身了解不够。如果你问它如何添加 MCP 或斜杠命令是做什么的，它无法回答。

我们可以把所有这些信息都放进系统提示中，但考虑到用户很少问这些问题，这会增加上下文负担并干扰 Claude Code 的主要工作：写代码。

相反，我们尝试了一种渐进式披露的形式。我们给了 Claude 一个文档链接，它可以加载并搜索更多信息。这样做有效，但我们发现 Claude 会在上下文中加载大量结果来找正确答案，而实际上你需要的只是答案本身。

所以我们构建了 Claude Code Guide 子 Agent，当你问关于 Claude Code 本身的问题时，Claude 会被提示调用它。这个子 Agent 有大量关于如何搜索文档以及返回什么的详细指令。

虽然这还不完美——当你问它如何设置自己时 Claude 仍然会困惑——但比过去好多了！

我们能够为 Claude 的行动空间增加内容，而不需要添加新的工具。

## 艺术，而非科学

如果你希望这是一组关于如何构建工具的严格规则，恐怕这不是你要找的指南。为模型设计工具既是科学也是艺术。它很大程度上取决于你使用的模型、Agent 的目标以及它所处的环境。

经常实验、阅读你的输出、尝试新事物。像 Agent 一样思考。
