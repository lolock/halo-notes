Markdown正从“人类友好的写作语法”演变为“AI最优的序列化协议”，而HTML则重新成为人类消费复杂信息的默认UI层。

最近构建了一个内容生产流水线：**Human -> Markdown -> LLM -> HTML -> Human (Reader)**。

具体来说，我使用Markdown撰写公众号文章初稿，由LLM将其转换为带有主题样式（如“秋日暖光”）的HTML片段，随后直接粘贴至公众号编辑器。读者最终消费的是经过语义编排与视觉增强的HTML。

这个流程中，一个角色关系发生了根本性变化：

- **Markdown** 不再是面向人类的最终呈现格式，而是**人与AI之间的通信协议**。
- **HTML** 不再是需要手动编写的复杂标记，而是**AI面向读者的渲染目标**。

一旦意识到这一点，就无法回到“Markdown优先为人”的旧认知。

**为什么Markdown更适合AI，HTML更适合人？**

从语言模型处理的角度分析，Markdown具备以下工程优势：

- **Token效率高**：同等语义下，Markdown比富文本或HTML的token消耗更低。
- **结构低负担**：以#、-、1.等轻量标记表示层级，解析成本极低。
- **语义明确**：标题、列表、引用、代码块有明确边界，便于分段、续写、切片和embedding。
- **Diff友好**：版本控制和增量修改非常清晰。
- **RAG友好**：向量化时，自然语言+轻标记的组合比混入大量标签噪声更稳定。

因此，Markdown正逐渐成为**LLM的事实中间语言**，尤其适合作为prompt、few-shot示例和工具调用的上下文格式。

**HTML的人性优势**

当信息量增大时，纯Markdown暴露出对人阅读不友好的问题：

- **无视觉锚点**：缺乏排版带来的层次感，用户需要自行进行注意力分拣。
- **无信息边界**：所有内容线性排列，无法快速区分主次、补充或元信息。
- **无交互能力**：无法折叠、展开、悬浮提示、异步加载。

而HTML天然具备三个层面的能力，正是大信息量下人类需要的：

- **语义层**：<article>、<section>、<aside> 表达内容角色。
- **结构层**：嵌套与盒子模型形成信息层级和视觉距离。
- **交互层**：<details>/<summary>、<dialog>、悬停等组件实现可控的信息暴露。

简单说：**Markdown擅长“表达内容是什么”，HTML擅长“定义信息怎么被消费”**。

传统Web内容栈：

> 人写HTML / Markdown → 浏览器 / 渲染器 → 人读

AI介入后的新栈（以我的工作流为例）：

> 人写Markdown (作为意图) → AI理解与转换 → HTML (作为渲染目标) → 人读

这里的关键变化有两个：

1. **Markdown的使用者变了**：之前是人为了“少写标签”而用它；现在是AI为了“高解析效率”而需要它。
2. **HTML的生产者变了**：之前是人手写或模板生成；现在是AI动态生成，并且可以根据阅读场景做自适应。

回到最开始的公众号写作场景，我建议的工程实践是：

- **Markdown作为源语言**：人写作和AI理解的最优交集。
- **LLM作为编译器**：将Markdown编译为目标HTML，可注入主题、响应式、交互组件。
- **HTML作为发布产物**：直接粘贴到公众号或其他平台。

这个模式不只适用于文章，也适用于AI报告、数据分析、代码解释、知识库问答等一切信息量较大的输出场景。
